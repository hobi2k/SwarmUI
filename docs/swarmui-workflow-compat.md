# SwarmUI 워크플로우 호환성 수정 기록

작성일: 2026-04-08

## 개요

SwarmUI에서 원격 ComfyUI 백엔드 자체는 정상 연결되어 있었지만, 등록된 워크플로우 5개 중 2개만 실행에 실패하는 문제가 있었다.

- 성공: 3개
- 실패: 2개
  - 동영상 기본 워크플로우
  - 대량 와일드카드용 이미지 워크플로우

이 패턴은 네트워크, CORS, 백엔드 주소, ComfyUI 서버 다운 같은 전체 연결 문제가 아니라, 특정 워크플로우를 SwarmUI 프론트엔드가 복원하거나 실행용 프롬프트로 변환하는 과정에서 깨지는 문제일 가능성이 높다는 뜻이었다.

## 이번 문서에서 중점적으로 다루는 대상

이 문서는 우선 `동영상 기본 워크플로우`가 왜 실패했고, 프론트엔드에서 어디를 어떻게 바꿨으며, 그 결과 실제로 무엇이 달라졌는지를 상세하게 기록한다.

추가로, 대량 와일드카드용 이미지 워크플로우와 공통으로 영향을 줄 수 있는 방어 로직도 함께 정리한다.

## 동영상 기본 워크플로우의 문제 원인

동영상 기본 워크플로우는 일반 `.json`이 아니라 `.app.json` 계열 워크플로우였고, `extra` 아래에 다음 메타데이터를 포함하고 있었다.

- `extra.linearMode = true`
- `extra.linearData = { inputs, outputs, ... }`

이 메타데이터는 ComfyUI의 앱 모드 또는 선형 편집 UI 상태를 복원하기 위한 정보로 보이지만, SwarmUI가 임베디드한 Comfy 프론트에서는 이 값을 별도로 정리하거나 호환 처리하는 코드가 없었다.

즉, SwarmUI는 저장된 워크플로우 JSON을 거의 그대로 읽어서 다음 경로로 넘기고 있었다.

- 저장된 워크플로우 로드 시: `loadGraphData(...)`
- 생성된 워크플로우 import 시: `loadApiJson(...)`

이때 그래프 실행에 직접 필요하지 않은 앱 모드 메타데이터까지 함께 들어가면서, 특정 `.app.json` 워크플로우에서 프론트 복원 또는 실행 준비 과정이 깨질 가능성이 있었다.

## 프론트엔드에서 실제로 바꾼 내용

### 1. 워크플로우 전처리기 추가

다음 목적의 전처리 함수를 추가했다.

- SwarmUI에서 워크플로우를 불러오기 전에 앱 모드 전용 메타데이터 제거
- 실행과 무관한 임시 프론트 상태 제거
- 재저장 시 같은 문제가 다시 들어오지 않도록 저장 전에도 동일 정리 적용

추가된 핵심 함수는 다음과 같다.

- `comfySanitizeWorkflowValue(...)`
- `comfyPrepareWorkflowForSwarm(...)`

### 2. 동영상 기본 워크플로우에 직접 영향을 준 정리 로직

`comfyPrepareWorkflowForSwarm(...)` 안에서 다음 값을 제거하도록 했다.

- `workflow.extra.linearData`
- `workflow.extra.linearMode`

이 변경의 의미는 다음과 같다.

- 그래프의 노드/링크/입출력 구조는 유지
- 앱 모드 전용 UI 복원 정보만 제거
- SwarmUI는 실행 가능한 그래프만 남겨 Comfy 프론트에 전달

즉, `동영상 기본 워크플로우`는 더 이상 SwarmUI 안에서 앱 모드 메타데이터 때문에 깨지지 않도록 바뀌었다.

### 3. 적용 위치

이 전처리기는 아래 3곳에 적용했다.

1. 저장된 워크플로우를 불러올 때
   - `comfyLoadByName(...)`
2. Generate 탭에서 생성된 워크플로우를 Comfy 탭으로 가져올 때
   - `comfyImportWorkflow()`
3. 워크플로우를 저장할 때
   - `comfySaveModalSaveNow()`

저장 시에도 같은 정리를 하도록 만든 이유는, 한 번 정상화한 워크플로우를 다시 저장했을 때 문제 메타데이터가 계속 남아서 다음 로드 때 같은 장애를 반복하지 않게 하기 위해서다.

## 동영상 기본 워크플로우 수정 결과

현재 확인된 결과는 다음과 같다.

- `동영상 기본 워크플로우`는 이전처럼 로드 단계에서 바로 깨지지 않도록 수정되었다.
- 사용자가 직접 확인한 기준으로, 해당 워크플로우 부분은 수정된 상태라고 볼 수 있다.

즉 이번 패치의 직접적인 효과는 다음과 같다.

- SwarmUI가 `.app.json` 워크플로우를 읽을 때 앱 모드 메타데이터 때문에 실패하던 경로를 제거
- 동영상 기본 워크플로우를 SwarmUI 안에서 다룰 수 있는 형태로 정리
- 이후 저장 시에도 같은 문제 메타데이터가 재유입되지 않도록 차단

## 추가로 함께 넣은 방어 로직

동영상 기본 워크플로우 수정과는 별도로, 실행 시점에 다음 오류가 뜨는 경로도 확인되었다.

- `TypeError: Cannot read properties of undefined (reading 'inputs')`

이 오류는 SwarmUI가 임베디드 ComfyUI에서 멀티 GPU 분할 실행을 가로채는 로직 안에서, 특정 노드 ID가 항상 실행 프롬프트 `output` 안에 존재한다고 가정하던 부분에서 발생할 수 있었다.

문제 코드는 대략 이런 흐름이었다.

- 워크플로우의 output node를 색상 기준으로 그룹화
- 각 그룹별로 조상 노드를 재귀적으로 모아 새 프롬프트 생성
- 이때 `output[nodeId].inputs`를 바로 참조

하지만 일부 커스텀 워크플로우에서는:

- 워크플로우 그래프에는 node가 보이지만
- 실제 `graphToPrompt()` 결과의 `output` 객체에는 그 node가 직접 존재하지 않을 수 있다

이 경우 `output[nodeId]`가 `undefined`가 되면서, 결국 `undefined.inputs`를 읽으려다 예외가 터진다.

### 이 부분에서 바꾼 내용

멀티 GPU 분할 프롬프트 재구성 함수 안에 다음 방어 로직을 넣었다.

- `output[nodeId]`가 없으면 예외를 던지지 않음
- 콘솔에 경고를 남김
- 해당 워크플로우는 멀티 GPU 분할을 포기
- 원래의 일반 `queuePrompt(...)` 경로로 자동 폴백

즉, 이 수정의 목적은 다음과 같다.

- 특정 워크플로우가 분할 실행과 맞지 않아도 전체 실행 자체를 망치지 않기
- 프론트가 죽는 대신 정상 단일 큐 실행으로 넘어가게 만들기

## 결과적으로 현재 프론트엔드에서 보장되는 것

현재 패치 기준으로 SwarmUI 프론트는 다음을 수행한다.

- `.app.json`의 `linearData`, `linearMode`를 제거하고 로드
- stale한 `rgthree.compare._temp_*` 문자열을 정리
- 저장할 때도 같은 정리를 재적용
- 멀티 GPU 분할 중 prompt node 불일치가 생기면 예외 대신 일반 큐 실행으로 폴백

## 동영상 기본 워크플로우 기준으로 기대되는 사용자 체감 변화

패치 전:

- `.app.json` 기반 워크플로우가 SwarmUI에서 정상 복원되지 않거나
- 앱 모드 메타데이터 때문에 특정 워크플로우만 비정상 동작할 수 있었음

패치 후:

- 동영상 기본 워크플로우를 SwarmUI가 더 단순한 그래프 형태로 받아들임
- 앱 모드 메타데이터 의존성을 제거한 상태로 로드
- 재저장 시에도 같은 메타데이터가 계속 남지 않음

## 아직 남아 있는 가능성

동영상 기본 워크플로우의 `linearData/linearMode` 관련 문제는 이번 패치로 대응했지만, 아래와 같은 경우는 추가 대응이 필요할 수 있다.

- 다른 커스텀 노드가 자기 전용 프론트 상태를 `widgets_values` 등에 저장하는 경우
- ComfyUI Node 2.0 프론트와 구형 커스텀 노드가 충돌하는 경우
- 특정 영상 노드가 SwarmUI 임베디드 프론트에서만 다른 복원 로직을 타는 경우

즉 이번 수정은 `동영상 기본 워크플로우`에서 확인된 대표적인 프론트 호환성 문제를 해결한 것이고, 모든 커스텀 노드 상태 복원 문제를 완전히 끝낸 것은 아니다.

## 요약

이번 수정에서 `동영상 기본 워크플로우`에 대해 핵심적으로 한 일은 다음 3가지다.

1. `.app.json`의 `extra.linearData`, `extra.linearMode`를 SwarmUI 로드 전에 제거했다.
2. 같은 정리를 저장 시에도 적용해서 문제 메타데이터가 다시 저장되지 않게 했다.
3. 실행 중 멀티 GPU 분할 경로에서 `undefined.inputs`가 터질 수 있는 부분을 막고, 실패 시 일반 큐 실행으로 폴백하게 했다.

그 결과, 동영상 기본 워크플로우는 이전보다 SwarmUI 프론트에서 안정적으로 로드되고 유지될 수 있는 상태가 되었다.
